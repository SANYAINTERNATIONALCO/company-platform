# Project Architecture Report — منصة شركة سانيا الدولية (Company Platform)

_Generated by static analysis of the repository + live Supabase project (`idsedrnuopflzepasmvc`) on 2026-07-15. Read-only audit — no code or data was modified to produce this report._

---

# Executive Summary

This is a single-tenant, single-organization internal HR + Finance management platform for **Sanya International Company** (شركة سانيا الدولية), built for one active construction project ("Al‑Amir Cement Project"). It is a **Next.js 16 App Router application that is architecturally a single-page application**: there is exactly one real route (`/`) plus one API route (`/api/pdf`); the 13 functional modules (Employees, Attendance, Payroll, Documents, Custody, Contracts, Overtime, Finance, Receipts, Visa, Tasks, Reports, ActivityLog) are not Next.js pages — they are React components conditionally rendered inside `app/page.tsx` based on a `useState` string, not the router. All data access is direct **Supabase** (Postgres + Auth + Storage) calls from the browser using a hardcoded anon key; there is no custom backend business layer except one Puppeteer-based PDF-rendering endpoint.

The codebase is small (17 TypeScript/TSX files under `app/`), fully Arabic/RTL, hand-styled with inline React style objects, and was clearly built iteratively and pragmatically by a single developer (the HR manager) rather than a team following a component-library/design-system process. It works and is in active production use (167 activity-log rows, 609 attendance records, real employee data), but it carries meaningful architectural debt: **authorization exists only in the UI layer** (every Postgres table has a wide-open `USING (true)` RLS policy), there is heavy copy-paste duplication (15 independent Supabase client instantiations, 2 divergent Arabic number-to-words implementations, repeated Arabic month-name arrays, repeated letterhead-loading logic), and there is no code-splitting, so the entire app (all 13 modules + recharts + xlsx) ships in one JS bundle regardless of which module a user actually opens.

The standout piece of engineering in the codebase is `app/api/pdf/route.ts`: a genuinely careful, well-commented two-phase Puppeteer PDF pipeline that measures real rendered page counts (via `pdf-lib`) to decide whether a signature block needs to be pushed to a new page, rather than guessing — solving a real, previously-broken problem (documented in `CLAUDE.md`'s "critical rules" section and recent commit history).

### Legacy / dead schema (present in the database, not referenced anywhere in the app code)

- **`profiles`** table (FK to `auth.users`, `role` default `'employee'`) — 0 rows, unused. Appears to be a leftover from an earlier Supabase-starter-template scaffold before the app switched to `user_roles` + `employees`.

`employee-files`, `visa-files`, `receipt-attachments`, `signatures`, `assets` — all public, all with a broad `SELECT` policy that permits *listing* bucket contents (flagged by Supabase's advisor), not just fetching known URLs.

# Authentication

- **Provider:** Supabase Auth, email + password only (`supabase.auth.signInWithPassword`) — no social login, no magic link, no MFA.
- **Client:** `app/page.tsx` instantiates its own `createClient(url, anonKey)` (the URL and anon JWT are literal strings in source — intentional, per CLAUDE.md: "لا تغيّره" / "don't change it"). Session persistence relies on `supabase-js` defaults (browser `localStorage`), with no explicit session/refresh configuration in code.
- **Login flow:** `login()` calls `signInWithPassword`; on success it sets local `user` state and calls `logActivity('تسجيل دخول','auth', ...)`. **Failed logins are not logged**, and there is no client- or server-side rate limiting/lockout — the login form is a plain password form with no throttling.
- **Role resolution:** after a successful sign-in, `loadRole()` queries `user_roles` by `user.id`. If no matching row exists, the app silently assigns the literal string `'viewer'` as the role — a role that is **not** one of the 5 valid roles used anywhere else in the authorization logic (see below), which is a dead/unhandled edge case rather than a deliberate tier.
- **Logout:** `supabase.auth.signOut()` + local state reset.
- **Duplication:** the exact same `createClient(url, anonKey)` call is repeated **15 times** — once in `page.tsx`, once in `logActivity.ts`, and once in each of the 13 `components/*.tsx` files — instead of being centralized in a shared module. Functionally harmless (same singleton values), but a maintainability and consistency smell.
- **Known users (from `docs/database.md`):** `hr@sanyacement.com` → editor, `site.manager@sanyacement.com` → admin, `husseinsattar651@gmail.com` → accountant, `guest_1@sanyacement.com` → guest_1. New users must be created in Supabase Auth first, then manually inserted into `user_roles` via SQL — there is no in-app user-management UI.

# Authorization

Five roles, defined by a `CHECK` constraint on `user_roles.role`:

| Role | Access |
|---|---|
| `editor` (حسن، مدير الموارد البشرية) | Full read/write everywhere, including Activity Log |
| `admin` (مدير الموقع) | Read-only everywhere **except**: may upload their own signature on Payroll and Attendance monthly approval, sees all Tasks, can print saved Documents |
| `accountant` | Finance + Receipts + Tasks only |
| `guest_1` / `guest_2` | Read-only on everything they can see |

**Enforcement is UI-only.** `page.tsx` computes `isReadOnly = ['admin','guest_1','guest_2'].includes(userRole)` and passes a `readOnly` boolean prop into each section component, which then conditionally hides buttons/inputs/forms. A handful of components layer extra, bespoke role checks on top (e.g. `userRole === 'admin'` gates the site-manager signature slot in `Payroll.tsx`/`Attendance.tsx`; `userRole === 'editor'` gates the Activity Log nav item; `Tasks.tsx` encodes an assignment hierarchy — editor can task the accountant, admin can task anyone but themselves, accountant can task no one).

**There is no server-side or database-level authorization.** Because every Postgres table's RLS policy is `USING (true)`/`WITH CHECK (true)` for all roles (see Current Problems), the `readOnly` prop and role checks are UX conveniences, not security boundaries — anyone with the (client-bundled, therefore public) anon key can call the Supabase REST API directly and read/write/delete anything regardless of their assigned role, including inserting a row into `user_roles` to self-grant `editor`.

`Tasks.tsx` additionally hardcodes the 3 assignable users' UUIDs, emails, and roles directly in source (`loadAllUsers()`), rather than deriving them from `user_roles`/`auth.users` — functional today with 3 fixed staff members, but brittle (a UUID rotation or a new hire requires a code change + redeploy, not a database update).

# Major Modules

The app has 13 functional sections plus a Dashboard, all driven from `page.tsx`'s `navGroups`/`activeSection` state (not routes). Below is what each does; the requested "Employee / Attendance / Leave / Overtime / Payroll" modules are covered in dedicated sections next since they're the HR core, and the rest are summarized here.

- **Documents** (الكتب الرسمية) — generates 6 types of official bilingual (AR/EN) HR letters (salary certificate, monthly attendance letter, commencement, continuity, warning, termination/clearance), auto-numbered `HR-{TYPE}-{YEAR}-{NNN}` via `document_counters`, with full content frozen into `doc_content` JSONB so any past letter can be reprinted byte-for-byte later. Includes an in-app Arabic number-to-words converter for salary amounts, adjustable print font size, adjustable HR-signature scale, and shared letterhead/footer image management (also reused by Attendance and Payroll).
- **Custody** (العهد المالية) — tracks company property (vehicle/laptop/phone/tools/other) checked out to employees, with active/returned tabs, per-employee "how many active items" badges, and undo-return.
- **Contracts** (العقود) — fixed-term/permanent contract tracking with a 30-day expiry warning banner, renew (archives the old contract as `renewed` and pre-fills a new form) and terminate workflows, full archive tab.
- **Visa** (التأشيرات) — four sub-tabs: (1) foreign-worker headcount stats by nationality/category with per-cell Excel-file attachments; (2) tourist visas (30/60-day) with a client-side "sweep and flag violated" pass run on every page load; (3) annual visas (1-year, Postgres `GENERATED` expiry column) with a 120-day warning window; (4) a full exit/return "cycle" state machine (`grace_period → exit_visa_issued → departed → completed`, 60-day grace period, 10-day exit-visa window) supporting both individual people and named bulk groups.
- **Finance** (الحسابات) — cash-advance ("fund") ledger with an auto-computed remaining balance (`funds_summary` view), a low-balance alert (≤10% of received amount) surfaced both here and on the Dashboard, a unified view that merges plain expenses with all three receipt types per fund, Excel export, and `window.print()`-based reporting (intentionally not using the Puppeteer pipeline, per CLAUDE.md).
- **Receipts** (الوصولات) — three independently-sequenced receipt books (fuel/كاز `P-####`, maintenance/صيانة `C-####`, cash-delivery/تسليم `M-####`) via a `get_next_receipt_number()` RPC with a client-side manual-count fallback, optional file attachments, an Arabic amount-in-words converter for delivery receipts (implemented independently from Documents.tsx's version — see Current Problems), and `window.print()`-based single-receipt printing with 2–3 signature blocks depending on type.
- **Tasks** (المهام) — internal task assignment respecting a hardcoded 3-person hierarchy, "mine / created by me / all" tabs, unseen/overdue badges that also feed the Dashboard's alert banner.
- **Reports** (التقارير السنوية) — a year-scoped analytics dashboard: 4 `recharts` visualizations (monthly payroll bars, stacked monthly expense bars by source, attendance line chart, leave-type pie chart) plus a detailed monthly summary table, entirely computed client-side from raw row fetches (no server-side aggregation view, unlike Attendance).
- **ActivityLog** (سجل النشاطات, editor-only) — global audit trail (last 500 rows) with section/date filters, plus a separate "login history" tab sourced from the same `activity_log` table filtered to `section = 'auth'` (last 300).
- **Dashboard** (in `page.tsx`) — role-gated stat tiles (active employees, pending tasks, low funds, visa violators), task/visa alert banners, and a mini `recharts` bar chart of the current year's salaries vs. spending.

# Employee Module

`app/components/Employees.tsx`. Roster CRUD (name, job title, phone, hire date, legacy `salary` field, passport number, shift type `يومي`/`روتيشن`, active/inactive status). Selecting an employee opens a detail view with:
- A fixed 8-category **file cabinet** (ID card, bank account, employment contract, start-of-work, resignation, warning, passport, other) backed by the `employee-files` Storage bucket + `employee_files` table.
- A free-text **notes timeline** (`employee_notes`).
- A live **regular-leave balance** indicator (2 days/month, non-cumulative, only for `يومي` shift employees — computed by counting the current month's `'إجازة اعتيادية'`/legacy `'إجازة'` rows in `attendance_records`; `روتيشن` employees show a static "week off per month" badge instead).

The employee list is independently re-fetched (its own `loadEmployees()` `select('*')`) by **six other components** (Attendance, Payroll, Custody, Contracts, Documents, Overtime) — there is no shared employee cache/context, so each section pays its own round-trip and can show a momentarily stale list relative to a change just made elsewhere in the same session.

# Attendance Module

`app/components/Attendance.tsx` (the largest and most sophisticated component, ~1,027 lines). Two view modes:

- **Daily entry** (`viewMode='daily'`): a grid of all active employees for a selected date. Employees without a record for the date get a computed default: `يومي` shift → `'حاضر'`; `روتيشن` shift → carried forward from their **last known status** (fetched per-employee via `getLastRotationStatus`, run sequentially in a loop — see Performance). Inline editable status/check-in/check-out/notes per row, bulk "set all to X" dropdown, per-row and full-day delete, and a single "Save all changed" action that iterates changed employees and **upserts them one request at a time** (not a single batched upsert). Saving a status transition into/out of `'إجازة تعويضية'` automatically debits/credits `employees.overtime_leave_balance` by 1.
- **Monthly summary** (`viewMode='monthly'`): reads the `monthly_attendance_summary` Postgres view, multi-month and multi-employee selectable. Columns are driven by **one single array** (`summaryColumns`) that simultaneously controls the on-screen table, the "الأعمدة" show/hide picker, the Excel export, and the PDF report — genuinely good single-source-of-truth design, explicitly called out in CLAUDE.md. Supports a per-employee "detailed daily report" drill-down, Excel export (`xlsx`), and a landscape Puppeteer PDF with the real company letterhead and a dual sign-off workflow (`attendance_approvals`, `site_manager`/`hr_manager`, independent per month).

# Leave Module

**There is no dedicated "Leave" component or table** — leave is a first-class concept but it is implemented as a set of literal Arabic **status values** inside `attendance_records.status` (`'إجازة مرضية'`, `'إجازة طارئة'`, `'إجازة اعتيادية'`, `'إجازة تعويضية'`, `'إجازة وفاة'`, plus `'عطلة رسمية'` for public holidays), enforced by a Postgres `CHECK` constraint and never renamed per CLAUDE.md rule #3. Business rules per leave type:

- **Regular / اعتيادية**: 2 days/month, non-cumulative ("use it or lose it"), daily-shift employees only; enforced as a **warning, not a hard block** (`updateRecord` in `Attendance.tsx` alerts but still allows saving past the limit).
- **Rotation / روتيشن**: implicit — a full week off per month is baked into the rotation shift pattern rather than tracked as leave days.
- **Death / وفاة**: 5 paid days (business rule documented in CLAUDE.md; not separately broken out in the `monthly_attendance_summary` view — it's folded into the total-days column only).
- **Sick / مرضية**: unlimited.
- **Compensatory / تعويضية**: see Overtime Module below — this is the one leave type backed by a real ledger rather than just a status value.

All leave *reporting* surfaces through the Attendance monthly view and the Reports module's leave-type pie chart; there is no standalone leave-request/approval workflow (no "pending leave request" concept — leave is recorded directly as an attendance-day fact, after it happens, by HR).

# Overtime Module

`app/components/Overtime.tsx`. A **day-based (never hour-based, by explicit and repeated project rule)** ledger: each `overtime_records` row (`days_count`, optional `start_date`/`end_date` or neither, notes) immediately adds `days_count` to `employees.overtime_leave_balance` on insert, and refunds it on delete. This balance is a distinct pool from the regular-leave allowance and is consumed **1 day at a time** — regardless of how many days a given overtime record originally granted — whenever a single day is marked `'إجازة تعويضية'` in daily Attendance, and refunded by 1 if that day is later changed or deleted. The module itself shows a sortable balance table (all employees, highest balance first) and a searchable full ledger with delete (with an explicit confirm dialog stating the balance impact).

# Payroll Module

`app/components/Payroll.tsx`. Month-scoped (`YYYY-MM`) working view over all active employees:
- Base salary and advance-schedule fields (`base_salary`, `advance_total`, `advance_monthly_deduction`, `advance_remaining`, `advance_total_installments`, `advance_completed_installments`, `sort_order`) are inline-editable per employee.
- Absence count for the month is **queried live** from `attendance_records` (`status = 'غائب'`), not stored — `absent_deduction = round(base_salary/30 * absent_days)`.
- Advance deduction and any extra amount are per-employee editable **drafts** until "Save & archive" is pressed.
- **Archiving** (`saveMonthPayroll`) upserts one `payroll_records` row per employee (again, one request per employee in a loop, not batched) and, **only on an employee's first save for that month**, decrements `advance_remaining` and increments `advance_completed_installments` — explicitly guarded against double-counting on re-save.
- **Un-archiving** (`deleteArchive`) precisely reverses that same advance-balance effect for every employee that had a saved record, before deleting the month's `payroll_records` and `payroll_approvals` rows — a well-thought-out symmetric operation.
- Dual sign-off (`payroll_approvals`, `site_manager`/`hr_manager`, independent per month) gates a "بوث approved" banner; landscape Puppeteer PDF with letterhead, selectable subset-of-employees printing, and running totals row.

# PDF Generation

Fully implemented in `app/pdfPrint.ts` (thin client helper — POSTs `{contentHtml, signatureHtml, styleCss, headerImageUrl, footerImageUrl, landscape}` to `/api/pdf` and triggers a blob download) and `app/api/pdf/route.ts` (the actual engine). Used by Attendance (monthly report + per-employee detail report), Payroll, and Documents; **Finance and Receipts deliberately use the older `window.print()` approach instead** (explicit CLAUDE.md decision, not an oversight).

Key design points, verified in code:
1. **Real headless Chromium**, not `jsPDF`/canvas rasterization — `puppeteer` locally, `puppeteer-core` + `@sparticuz/chromium` on Vercel (selected via `process.env.VERCEL_ENV`).
2. **Deterministic font rendering**: the Cairo webfont is fetched once from Google Fonts, has its `woff2` URLs rewritten to inline `data:` URIs, and is cached in a module-level variable (`cachedFontCss`) — this avoids the font-swap-during-measurement bug that previously caused inconsistent page counts for identical content (explicitly documented in a code comment and in CLAUDE.md's critical-rules section).
3. **Real letterhead sizing**: header/footer images are fetched server-side, their true pixel dimensions read via `image-size`, and scaled to span the full printable width at their original aspect ratio (capped at 40mm/30mm as a safety net against a badly-proportioned uploaded image) — margins are then computed *from* the actual image height, not a guessed constant.
4. **Two-phase, measurement-driven signature placement** (the most sophisticated part of the codebase): the content table is rendered once *alone* to get its true page count via `pdf-lib`'s `getPageCount()`. If that's 1 page, the exact remaining pixel space is measured in-browser (`getBoundingClientRect()`) and the signature block is placed directly below the content if it fits, or pushed to a forced new page if it doesn't. If the content alone is multi-page, the content+signature combination is first rendered "naturally" (no forcing) and its page count compared against the content-alone count — only if the signature genuinely didn't fit within the natural flow is a forced page-break applied. This avoids the earlier bug (documented in CLAUDE.md rule #7 and recent commit history — `fix unnecessary blank final page for signature block`, `fix signature page-fit decision with ground-truth page counting`) where even a few pixels of trailing content (e.g. a "generated by..." footer line) could push an entire signature block onto an otherwise-blank extra page.
5. `next.config.ts` requires an explicit `outputFileTracingIncludes` entry for `@sparticuz/chromium/bin/**/*` — without it, Vercel's file-tracing doesn't pick up the dynamically-loaded Chromium binary and PDF generation fails in production with "input directory does not exist" (documented gotcha in CLAUDE.md rule #7).

Caveat: the `cachedFontCss` cache is **process-local** (in-memory), so it does not survive across separate serverless function instances — a cold start (which `maxDuration = 30` and Vercel's scale-to-zero behavior make routine) re-fetches Google Fonts synchronously inside the request, adding latency to the first PDF generated by a fresh instance.

# Reporting

Two distinct reporting surfaces:
1. **`Reports.tsx`** — a year-scoped analytics dashboard (payroll totals, expense breakdown by source, attendance line chart, leave-type distribution pie chart, plus a full monthly table) computed **entirely client-side** from raw row fetches across `payroll_records`, `expenses`, `fuel_receipts`, `maintenance_receipts`, `delivery_receipts`, and `attendance_records` for the selected year — there is no server-side aggregation (unlike Attendance, which benefits from the `monthly_attendance_summary` view).
2. **Per-module printed reports** — Attendance's monthly/detailed PDF, Payroll's payroll-sheet PDF, Finance's fund/summary `window.print()` reports, and Receipts' single-receipt `window.print()` slips. These are export/print artifacts rather than interactive reporting, but functionally serve as the "official" reports handed to management.

There is no scheduled/emailed reporting, no report-builder, and no cross-module combined report beyond what `Reports.tsx` and the Dashboard's mini-chart already surface.

# Settings

**There is no dedicated Settings module or screen.** Configuration-like functionality that would typically live in a Settings area is instead scattered across the modules that need it:
- **Letterhead/footer images** (`document_assets`, keys `letterhead_top`/`letterhead_bottom`) are uploaded from **inside Documents.tsx**, and independently re-fetched (read-only) by **Attendance.tsx and Payroll.tsx** via their own copies of `loadLetterheads()`.
- **Permanent HR signature** (for official Documents) is uploaded from inside Documents.tsx, stored in the `signatures` table.
- **Monthly approval signatures** (Attendance, Payroll) are uploaded inline within those respective modules' UI, not a shared settings surface.
- **Visa category counts** (`visa_stats`) are edited inline within the Visa module's stats tab.
- **User/role management** has **no UI at all** — adding a user is a manual two-step process (create in Supabase Auth, then `INSERT INTO user_roles` via SQL), documented in `docs/database.md` for whoever operates the Supabase dashboard directly.

# Components

All 13 module components plus `page.tsx` share a consistent but entirely hand-rolled visual language: white rounded cards (8–14px radius) on a `#f0f4f8` background, a `#1e40af`/`#3b82f6` blue primary palette, semantic color-pair status pill badges (green/red/amber/blue backgrounds with matching darker text), and RTL Arabic layout throughout (`direction: 'rtl'`) with a `system-ui` font stack for on-screen UI (separate from the Cairo font used only in generated PDFs).

There is **no shared component library** — every button, text input, select, table, badge, and dropdown is a literal inline JSX element with its own `style={{...}}` object, independently re-typed in every file that needs one. Recurring, effectively-duplicated UI patterns include:
- An `inputStyle`/`inputBase` object (padding/border/radius/font-size) redefined near-identically in ~10 of the 13 components.
- Custom "click outside to close" dropdown menus (month picker and employee picker in `Attendance.tsx`, receipt-type picker in `Receipts.tsx`) each independently wiring up a `useRef` + a `mousedown` document listener rather than sharing one `useDropdown` hook.
- A large hand-drawn inline SVG icon switch (`iconSvg()` in `page.tsx`) rather than an icon library or a shared `<Icon>` component.

`recharts` (Dashboard mini-chart, full Reports page) is the only external UI/visualization library actually used; Tailwind CSS is installed but not used by any component (see Current Problems).

# Shared Utilities

Nominally shared, but in practice thin and incompletely shared:
- **`app/logActivity.ts`** — the one genuinely shared utility: `logActivity(action, section, details)` looks up the current user + their role and inserts a row into `activity_log`. Called consistently from Employees, Attendance, Payroll, Documents, Custody, Contracts, Overtime, Visa, Receipts, Finance for sensitive operations, per CLAUDE.md rule #4. Fails silently (empty `catch`) so a logging failure never blocks the underlying operation.
- **`app/pdfPrint.ts`** — `generatePdf()` (calls `/api/pdf`, downloads the blob) and `esc()` (HTML-escapes a value for safe interpolation into the generated PDF's HTML — used consistently everywhere Puppeteer HTML is built).

Everything else that *should* be shared is duplicated instead (see Current Problems → Duplicated Code): Supabase client construction, Arabic month-name arrays/`monthLabel()`, Arabic number-to-words converters, letterhead-loading logic, and per-component input styling.

# APIs

The application exposes exactly **one custom HTTP endpoint**:

- **`POST /api/pdf`** (`app/api/pdf/route.ts`, Node runtime, `maxDuration = 30`) — accepts pre-built HTML fragments (content, signature block, CSS, optional header/footer image URLs, orientation) and returns a rendered PDF (`application/pdf`) using the two-phase Puppeteer pipeline described above. Stateless, no auth check of its own beyond whatever called it client-side already having a Supabase session — **the endpoint itself does not verify the caller's identity or role**, so it will render and return a PDF for any POST body regardless of who sends it, relying entirely on the fact that the URL isn't publicized (security-through-obscurity, not enforcement).

Every other "API" the app has is **direct Supabase access from the browser**: PostgREST (table CRUD), Storage (file upload/public-URL), Auth (sign-in/sign-out/session), and one Postgres RPC (`get_next_receipt_number`). There is no internal REST/GraphQL API layer, no server actions, and no middleware.

# Current Problems

### Security
1. **RLS is effectively disabled on all ~26 application tables.** Every table has a single `"Allow all"` policy (`USING (true)`/`WITH CHECK (true)`) for every command, to every role. Combined with the anon key being embedded directly in the shipped client bundle (an explicit, documented project decision), **anyone who extracts that key can read, write, or delete any row in any table directly via the Supabase REST API** — including salaries, passport numbers, signature images, and `user_roles` itself (i.e., a client could `INSERT` a row granting themselves `editor`). The entire role-based UI described in "Authorization" is a UX layer, not a security boundary.
2. **Three views are `SECURITY DEFINER`** (`funds_summary`, `monthly_attendance_summary`, `fund_receipts_summary`) — flagged `ERROR` by Supabase's own linter. Low marginal risk today (because the underlying tables are already open per #1) but blocks ever tightening RLS later without breaking these views, since they'd keep running with the creator's privileges rather than the querying user's.
3. **Storage buckets allow listing, not just fetching.** `employee-files`, `receipt-attachments`, `signatures`, `visa-files` are public buckets with a broad `SELECT` policy on `storage.objects`, meaning their entire contents can be enumerated (not merely retrieved by a known URL) — e.g. every employee's uploaded passport scan or every collected signature image.
4. **`get_next_receipt_number()`** has a mutable `search_path` (schema-injection-prone pattern, flagged by the linter).
5. **`rls_auto_enable()`** is a `SECURITY DEFINER` function directly callable via RPC by both `anon` and `authenticated` — unreferenced by any code read during this audit, so it's most likely leftover tooling that needlessly widens the attack surface.
6. **Leaked-password protection is disabled** in Supabase Auth (no HaveIBeenPwned check on new passwords).
7. **No login throttling/lockout** — `login()` is a bare `signInWithPassword` call with no attempt counting, and failed logins aren't even written to `activity_log` (only successes are).
8. **The `/api/pdf` endpoint performs no authorization check** of its own — it will render and return a PDF for any well-formed POST body.

### Data model
9. **Two generations of schema coexist.** `profiles`/`attendance` (legacy Supabase-starter tables, 0 rows, unreferenced) sit alongside the actively-used `employees`/`attendance_records`. `employees.salary` (legacy, still written by the Employees form) coexists with `employees.base_salary` (the one Payroll actually uses) — nothing keeps them in sync, so the two fields can silently diverge. `employees.advance_remaining_installments` is dead; the app uses `advance_total_installments`/`advance_completed_installments` instead.
10. **Missing indexes on ~13 foreign key columns** (Supabase performance advisor): `attendance.employee_id`, `contracts.employee_id`, `custody_items.employee_id`, `employee_files.employee_id`, `employee_notes.employee_id`, `expenses.fund_id`, `official_documents.employee_id`, `overtime_records.employee_id`, `tasks.assigned_to`, `tasks.created_by`, and the three `*_receipts.fund_code` columns. These are exactly the columns hit by every "load this employee's records" query used throughout Employees/Attendance/Payroll/Custody/Contracts.

### Duplicated code
11. **The Supabase client (`createClient(url, anonKey)`) is instantiated independently 15 times** — once per component plus `page.tsx` and `logActivity.ts` — instead of one shared module export.
12. **Arabic number-to-words is implemented twice, independently, with different vocabulary and different maximum magnitude** — `Documents.tsx`'s `numberToArabicWords()` handles up to billions and returns a bare string; `Receipts.tsx`'s version of the same name only handles up to millions and appends `"دينار عراقي فقط لا غير"` itself. Any future fix to one will not propagate to the other.
13. **Arabic month names / `monthLabel()`** are redefined independently in `Attendance.tsx`, `Payroll.tsx`, `Reports.tsx` (as `MONTHS_AR`), and again inline in `page.tsx`'s dashboard mini-chart (as `MONTHS_SHORT`, using abbreviations) — four separate literal arrays that must be kept manually in sync.
14. **Letterhead-loading (`loadLetterheads()` reading `document_assets`)** is copy-pasted verbatim into Attendance.tsx, Payroll.tsx, and Documents.tsx — each independently re-fetching the same 2 rows.
15. **Input styling objects** (`inputStyle`/`inputBase`) are redefined near-identically in roughly 10 of the 13 components.

### Performance
16. **No code-splitting.** All 13 section components, `recharts`, and `xlsx` are statically imported at the top of `page.tsx` and therefore bundled into the single initial JS payload — a user who only ever opens Attendance still downloads and parses Reports' charting code, Finance, Documents, etc.
17. **Sequential, unbatched writes in save loops.** `Attendance.tsx`'s `saveAll()` and `Payroll.tsx`'s `saveMonthPayroll()`/`deleteArchive()` each issue **one Supabase request per changed/affected employee inside a `for` loop**, rather than a single batched `upsert([...])` call — an N-employee save is N sequential network round trips.
18. **Sequential per-employee lookups in daily attendance load.** `loadDailyRecords()` calls `getLastRotationStatus()` inside a loop (not `Promise.all`'d) for every rotation employee lacking a record for the viewed date.
19. **Client-side "sweep and flag violated" pass on Visa tables** — `loadTouristVisas()`/`loadAnnualVisas()` loop over every fetched row on **every page load** and issue an individual `UPDATE` for any newly-expired visa, rather than a generated column, a scheduled job, or computing "violated" purely at render time without writing.
20. **Reports.tsx has no server-side aggregation** — it pulls a full year's raw rows from `attendance_records`/`payroll_records`/`expenses`/receipts tables into the browser and aggregates client-side, unlike Attendance's use of a dedicated Postgres view.
21. **Redundant per-mount data fetches with no cross-component cache** — the active employee list is independently fetched by 6+ components in a single session; letterhead images are independently fetched by 3 components; nothing is shared via context, so navigating between sections repeatedly re-fetches identical data.

### General maintainability
22. **`Tasks.tsx` hardcodes 3 users' UUIDs, emails, and display names directly in source** instead of deriving them from `user_roles`/`auth.users`.
23. **`app/layout.tsx` still has default `create-next-app` boilerplate** (`title: "Create Next App"`, matching description) — never customized to the actual product name.
24. **Tailwind CSS + PostCSS are installed as dependencies but never used** by any component (100% inline `style={{}}` objects) — dead weight in the dependency tree and build config.
25. **No automated tests** anywhere in the repository (no test runner configured, no `__tests__`/`*.spec.*` files), and **no CI configuration** found.
26. **Two parallel PDF/print systems** — the sophisticated Puppeteer pipeline (Attendance, Payroll, Documents) vs. the simpler `window.print()` approach (Finance, Receipts) — a deliberate, documented split (CLAUDE.md), but it means two independent print code paths with different fidelity guarantees must be maintained going forward.

# Suggested Improvements

1. **Tighten RLS to match the app's actual role model.** At minimum, restrict write operations on sensitive tables (`employees`, `payroll_records`, `user_roles`, `activity_log`) to `authenticated` users, and ideally encode the 5-role model (`editor`/`admin`/`accountant`/`guest_1`/`guest_2`) as real Postgres policies (e.g., via a `SECURITY INVOKER` helper function reading `user_roles`) so the database — not just the UI — enforces who can write what. This is the single highest-leverage change available.
2. **Switch the three `SECURITY DEFINER` views to `SECURITY INVOKER`** (Postgres 15+ supports this natively) once the underlying tables have real RLS, so the views don't silently bypass it.
3. **Add the ~13 missing foreign-key indexes** flagged by the Supabase performance advisor — a low-risk, high-value change (`CREATE INDEX` statements only).
4. **Turn off public "list" access on Storage buckets** (keep `SELECT` scoped to fetching a known object, not enumerating the bucket) unless directory listing is actually a required feature.
5. **Enable leaked-password protection** in Supabase Auth settings (a one-click toggle, no code change).
6. **Add basic login throttling** (Supabase Auth has built-in rate limiting options; at minimum log failed attempts to `activity_log` alongside successes for visibility).
7. **Retire the dead schema**: drop or explicitly archive `profiles`, `attendance`, `employees.salary`, and `employees.advance_remaining_installments` once confirmed unused, to stop the schema from misleading future readers (including future AI assistants working from `docs/database.md`).
8. **Consolidate the Supabase client into one module** (`app/lib/supabaseClient.ts`) and import it everywhere — purely mechanical, zero behavior change, removes 14 duplicate literals.
9. **Extract shared constants/utilities**: one `MONTHS_AR`/`monthLabel()`, one `numberToArabicWords()`, one `loadLetterheads()` (ideally via a small shared hook or context so it's fetched once per session, not per component mount), one `inputStyle`.
10. **Batch the save loops**: replace the per-employee `for` loops in `Attendance.saveAll()` and `Payroll.saveMonthPayroll()`/`deleteArchive()` with a single `upsert()` call passing an array of rows.
11. **Move the Visa "violated" status computation out of the read path** — either a Postgres `GENERATED`/computed check at query time (comparing `expiry_date` to `now()` without writing), or a scheduled Edge Function, instead of a client-side sweep-and-write on every tab open.
12. **Introduce code-splitting** for the 13 section components via `next/dynamic`, so a user who opens Attendance doesn't pay for Reports' `recharts` bundle and vice versa.
13. **Remove the unused Tailwind/PostCSS dependencies**, or commit to actually adopting them for new work, rather than leaving them installed-but-dark.
14. **Give `layout.tsx` real metadata** (title/description matching the actual product) — a 2-line fix.
15. **Consider a minimal shared UI kit** (Button/Input/Badge/Card/Dropdown as small components taking the already-consistent style values as props) to stop re-typing the same style objects — this could be introduced incrementally, file by file, without a big-bang rewrite.

# Recommended Refactoring

Given this is a small, actively-used, single-developer-maintained production app, the highest-value refactors are the **low-risk, mechanical ones that don't touch business logic**, prioritized roughly in this order:

1. **Extract `app/lib/supabaseClient.ts`** and update all 15 call sites to import the singleton instead of constructing their own client. Zero behavioral risk, immediate reduction in duplicate surface area.
2. **Extract `app/lib/dateLabels.ts`** with a single `MONTHS_AR` array and `monthLabel()` function; update Attendance/Payroll/Reports/page.tsx's dashboard chart to import it instead of maintaining 4 copies.
3. **Extract `app/lib/arabicNumbers.ts`** with one canonical `numberToArabicWords()` (reconciling the two current implementations to pick the more complete one, from `Documents.tsx`, which handles billions), and update `Receipts.tsx` to use it — this closes a real correctness-divergence risk, not just a style one.
4. **Extract a `useLetterheads()` hook** (or a small `DocumentAssetsContext`) that fetches `document_assets` once and is shared by Attendance/Payroll/Documents, replacing 3 copies of `loadLetterheads()`.
5. **Batch the payroll/attendance save loops** into single `upsert()` calls — this is a meaningful UX win (fewer round trips = faster saves) as well as a code-quality one, and is contained entirely within `saveAll()`/`saveMonthPayroll()`/`deleteArchive()`.
6. **Write a real RLS policy set** scoped to the existing `user_roles` table before the next time this app's data footprint or user count grows — this is the one item on this list that isn't "nice to have," since the current posture means the entire access-control model is advisory. It can be rolled out table-by-table (start with `payroll_records`, `employees`, `user_roles` — the highest-sensitivity tables) without a full rewrite.
7. Only after the above: consider whether the app's growth trajectory justifies moving section-switching from `useState` to real Next.js routes (`/employees`, `/payroll`, etc.) for deep-linking and code-splitting — this is a larger, more architecturally invasive change and should be the last, not first, item tackled, since the current single-page approach is not itself broken, just non-idiomatic for the framework it's built on.

None of the above requires touching the PDF generation pipeline (`app/api/pdf/route.ts`), which is already well-engineered and should be left alone beyond routine maintenance.
